Intelligent Timetable & Classroom Assignment System    

 Context

 Final year university project. The goal is to build a web-based tool that allows university administrators to input data (classrooms, professors, subjects, student groups) and automatically generate an optimal
  timetable that assigns classrooms, time slots, and professors while respecting constraints (no double-booking of professors, classrooms, or student groups).

 Tech stack: HTML/CSS/JS frontend | Node.js + Express backend | SQLite database
 Scope: University-level (faculties, degree programs, years, groups)
 Auth: Role-based (admin, viewer)
 Timeline: 3-4 months

 1.2 Database Schema

 -- Authentication
 CREATE TABLE users (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     username TEXT UNIQUE NOT NULL,
     password_hash TEXT NOT NULL,
     role TEXT NOT NULL CHECK(role IN ('admin', 'viewer')),
     created_at DATETIME DEFAULT CURRENT_TIMESTAMP
 );

 -- Core entities
 CREATE TABLE faculties (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     name TEXT NOT NULL UNIQUE
 );

 CREATE TABLE degree_programs (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     name TEXT NOT NULL,
     faculty_id INTEGER NOT NULL REFERENCES faculties(id) ON DELETE CASCADE
 );

 CREATE TABLE student_groups (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     name TEXT NOT NULL,                    -- e.g. "CS Year 2 Group A"
     year INTEGER NOT NULL,                 -- academic year (1, 2, 3...)
     num_students INTEGER NOT NULL,
     degree_program_id INTEGER NOT NULL REFERENCES degree_programs(id) ON DELETE CASCADE
 );

 CREATE TABLE professors (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     name TEXT NOT NULL,
     email TEXT UNIQUE,
     max_hours_per_week INTEGER DEFAULT 20
 );

 CREATE TABLE classrooms (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     name TEXT NOT NULL,                    -- e.g. "Room 101", "Lab A3"
     building TEXT,
     capacity INTEGER NOT NULL,
     type TEXT NOT NULL CHECK(type IN ('lecture', 'lab', 'both'))
 );

 CREATE TABLE subjects (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     name TEXT NOT NULL,
     type TEXT NOT NULL CHECK(type IN ('lecture', 'lab', 'seminar')),
     hours_per_week INTEGER NOT NULL,       -- how many hours per week
     degree_program_id INTEGER NOT NULL REFERENCES degree_programs(id) ON DELETE CASCADE,
     year INTEGER NOT NULL                  -- which year takes this subject
 );

 CREATE TABLE time_slots (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     day TEXT NOT NULL CHECK(day IN ('Monday','Tuesday','Wednesday','Thursday','Friday')),
     start_time TEXT NOT NULL,              -- "08:00"
     end_time TEXT NOT NULL,                -- "09:00"
     slot_order INTEGER NOT NULL            -- for sorting (1, 2, 3...)
 );

 -- Relationships
 CREATE TABLE professor_subjects (
     professor_id INTEGER NOT NULL REFERENCES professors(id) ON DELETE CASCADE,
     subject_id INTEGER NOT NULL REFERENCES subjects(id) ON DELETE CASCADE,
     PRIMARY KEY (professor_id, subject_id)
 );

 CREATE TABLE professor_availability (
     professor_id INTEGER NOT NULL REFERENCES professors(id) ON DELETE CASCADE,
     time_slot_id INTEGER NOT NULL REFERENCES time_slots(id) ON DELETE CASCADE,
     available INTEGER NOT NULL DEFAULT 1,  -- 1 = available, 0 = unavailable
     PRIMARY KEY (professor_id, time_slot_id)
 );

 -- Generated schedule
 CREATE TABLE schedules (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     name TEXT NOT NULL,                    -- e.g. "Fall 2026 v1"
     created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
     fitness_score REAL                     -- algorithm quality score
 );

 CREATE TABLE schedule_entries (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     schedule_id INTEGER NOT NULL REFERENCES schedules(id) ON DELETE CASCADE,
     subject_id INTEGER NOT NULL REFERENCES subjects(id),
     professor_id INTEGER NOT NULL REFERENCES professors(id),
     classroom_id INTEGER NOT NULL REFERENCES classrooms(id),
     time_slot_id INTEGER NOT NULL REFERENCES time_slots(id),
     group_id INTEGER NOT NULL REFERENCES student_groups(id)
 );

 Phase 4: Scheduling Algorithm (Week 6-9) -- Core of the project

 4.1 Problem Definition

 This is a Constraint Satisfaction Problem (CSP). Each schedule entry is an assignment of:
 (subject, professor, classroom, time_slot, group)

 4.2 Hard Constraints (must never be violated)

 1. No professor double-booking - a professor can only be in one place at a time
 2. No classroom double-booking - one class per classroom per time slot
 3. No group double-booking - a student group can only attend one class at a time
 4. Capacity - classroom capacity >= group's number of students
 5. Room type match - labs in lab rooms, lectures in lecture halls
 6. Professor qualification - professor must be assigned to teach the subject
 7. Professor availability - professor must be available at the assigned time slot
 8. Hours fulfilled - each subject must have exactly hours_per_week slots assigned

 4.3 Soft Constraints (optimize for, scored with fitness function)

 1. Minimize idle gaps in a group's day
 2. Minimize idle gaps in a professor's day
 3. Distribute classes evenly across the week (avoid overloaded days)
 4. Prefer keeping same-subject hours on different days (not all on Monday)

 4.4 Recommended Algorithm: Greedy Construction + Local Search

 Step 1 - Greedy construction:
 1. List all required assignments: for each (subject, group) pair, create N entries where N = hours_per_week
 2. Sort by "most constrained first" (subjects with fewer valid professor/room/slot options go first)
 3. For each assignment, try all valid (professor, classroom, time_slot) combinations
 4. Pick the one with the best fitness score that violates no hard constraints
 5. If no valid assignment exists, backtrack to the last successful assignment and try alternatives

 Step 2 - Local search improvement (optional but recommended):
 1. Take the constructed schedule
 2. Repeatedly try random swaps of two entries
 3. If a swap improves the fitness score without violating hard constraints, keep it
 4. Stop after N iterations without improvement

 4.5 Key Implementation Files

 - server/algorithm/constraints.js:
   - checkHardConstraints(schedule, newEntry) - returns true/false
   - Individual check functions: isProfessorFree(), isClassroomFree(), isGroupFree(), etc.
 - server/algorithm/fitness.js:
   - calculateFitness(schedule) - returns a numerical score (higher = better)
   - Individual scoring functions for each soft constraint
 - server/algorithm/scheduler.js:
   - generateSchedule(data) - main entry point
   - Takes all entities from DB, returns a complete schedule
   - Should log progress and return statistics (conflicts resolved, fitness score, etc.)
